## 0. このゲームは何か(前提共有)

ターン制・グリッドベースの思考パズル。詰将棋を目標体験とする(短手数・全要素に意味・解の一意性・妙手)。

- 画面に**2つの部屋(別々のグリッド)を並列表示**する
- 各部屋に**キャラ(ボール)が1体ずつ**いる
- プレイヤーの入力(上下左右)は**全キャラに同時に適用される**(完全同期・放送入力)
- **壁や外周にぶつかったキャラはその手をスキップし、もう片方は動く**。これが唯一のズレ発生源であり、パズルの核
- 「わざと壁に向かって歩く」=見えている部屋では無意味な1手が、もう片方の部屋では相対位置を調整する決定的な手(=**妙手**)になる
- **全キャラが同一ターンに同時にゴール到達でクリア(厳格ルール)**

この「別グリッド × 同時到達 × ターン制」の組み合わせは先行事例調査で未踏。詰将棋的な品質保証(解一意・手数固定・妙手保証)も既存のインディー同種作には存在せず、ここが差別化の主軸。

## 1. 確定したルール(リリース版の核)

| 項目 | 決定 |
|---|---|
| 部屋数 | **2部屋**(リリース版は固定。3枚版は拡張で対応、後述) |
| 表示 | **並列表示**(両部屋を常時画面に出す)。注視選択(1部屋のみ表示)は第二段階に保留 |
| 入力 | 上下左右。全キャラに同報 |
| ズレ発生源 | **壁・外周によるスキップのみ**(穴・その他ギミックは入れない) |
| クリア条件 | **厳格ルール = 全キャラが同一ターンに同時ゴール到達** |
| ゴール進入制限 | **同時でないゴール進入は反則(ミス)**(下記) |
| 盤上のオブジェクト | 壁・キャラ・ゴールの3種のみ(穴なし) |

### ゴール進入制限(確定・logicVersion 1.2〜)

**ゴールは同時到達の瞬間にしか踏めない。** あるキャラが自分のゴールへ入る手でも、その同じ手で全キャラが同時にゴールへ入るのでなければ、**その瞬間に反則(ミス)**となる。途中でゴールを通り道に使うことはできない。

- ゴールは壁ではない(=スキップさせて足踏みに使う対象ではない)。非同時にゴールへ入る手は「指せるが負け」=反則
- **ソルバー**: 非同時のゴール進入が起きる手は解の経路に含めない(反則手として枝刈り)。よって有効解は「ゴールには全員同時のときだけ踏み込む」もののみ
- **プレイ側**: プレイヤーが非同時にゴールへ入ったら、ボールは一旦ゴールへ「行って」から反則と判定し、**「一手戻す」か「初形へ」を選ばせる**(行って戻る/リスタート)
- 不変条件: 有効解では、最終手より前にどのキャラもゴールに乗らない。全員が同時にゴールへ入る最終手だけが終局(クリア)
- 効果: 同時到達のパリティ制約がさらに締まり、解数が減って一意問題が出やすい。検証: seeds 1-1000×4x4/5x5/6x6 で 2453問→1799問、解数最大も低下(例 4x4 で 41→21)
- ※検討の経緯: 一時「ゴールを条件付き壁にしてスキップ=足踏みを正規手段にする」案(1.1)も実装したが、ゴール前の足踏みが妙手として濫用でき意図と違ったため、**反則=ミス判定(1.2)に確定**

### 緩和ルールについて
プロトタイプには緩和ルール(ゴール到達でその場に固定)も実装したが、**リリース版は厳格ルール一本**で進める。理由: 厳格ルールの方が同時到達のパリティ制約で解が絞られ、詰将棋らしい一意な問題が出やすい(検証で一意率が緩和の約3倍)。

## 2. 難度設計の方針(検証済みの結論)

100問サンプル検証から得た重要な知見。**実装者はこの前提を覆さないこと。**

### 2-1. 難しさを決めるのは「手数」と「解の数」。盤面サイズではない
- 同じ手数帯で比較すると、4x4でも8x8でも最短解の数(=緩さ)はほぼ同じ(中央値2〜3で横並び)
- **盤面サイズは手数を経由して間接的に効くだけ**。サイズ自体は独立した難度軸にならない
- したがって**難度はサイズではなく、手数と解数フィルタで管理する**

### 2-2. 升目拡張の位置づけ
- サイズを上げること自体に「直接の難度上げ」のうまみはない(同手数なら同難度)
- ただし**長手数の問題を安定供給する「器」としての価値はある**。小盤では長手数の問題が物理的に作りにくい(空間が詰まる)
- 結論: **リリース版はサイズを看板にしない。難度は手数で見せる。** 升目を難度の独立軸として売るのは、注視選択(第二段階・視界制限)を載せて「広い盤=覚えきれない=難しい」が成立してから

### 2-3. 解数の暴走を防ぐ
- サイズ・手数を上げると最短解の本数が増え、「どう動いても詰む」緩い問題が混入する(8x8では最大2000解超の例も観測)
- これは**ソルバーの解数フィルタで制御する**。壁密度調整では抑えきれない
- 当初は「章が進むほど解数フィルタを厳しく(序盤 解≤3 → 中盤以降 解一意必須)」とする設計だったが、**難度の主軸を episodes に移したため解の一意性はハード条件にしない(下記 2-4)**

### 2-4. 手数より「ズレ調整エピソード数」が難度の主軸(プール実データで確認・2026-06-14)
- 手数が長くても「同じ壁に当て続けるだけ」の問題は易しい。難しさは**当て直す回数=ズレ調整エピソード数(episodes)**で決まり、手数はその結果として伸びる(因果の向き: episodes → 手数)
- エピソード = 解の経路で連続する壁当て手のかたまり。プレイヤーが要する「気づき」の近似単位
- 実証(全1799問): 同じ episodes 内で手数は大きく散る(episodes=1 でも手数 2〜19)。長手数でも episodes≤1 が41問、短手数でも episodes≥3 が6問。手数は episodes のノイズの多い代理にすぎない
- **多解でも可。難度は最易解で決まる**ので episodes は「最短解の中で最小になるエピソード数」を採る(多解1088問のうち25%で、見栄え重視の焼き込み経路は真の最易解より過大だった)
- 実装: **2段階構造** = 章(外側)が episodes、章内(内側)が サイズ→手数 の昇順。盤は 4x4→5x5 と広がり各サイズ内で手数が滑らかに増える。章境界の手数リセットは「新章の自然な再上昇」として吸収され、見た目(手数)と難度の逆行が消える。**視認性の限界から 6x6 は本体除外(5x5まで)**、導入(第一章の頭)は ep0 の 1〜2手のみ。全5章: 手ほどき(ep0-1)/ 二〜五つの綻び(ep2-5)。**ep6以上は在庫僅少のため MVP 除外**(プールには残し将来のアップデート章へ)。プールは 6x6 除外・seeds 1-6000 で再生成し ep5 を20問確保。3部屋は EXTRA章(別生成)。値は `analysis.episodes` に焼き込み済み(`tools/core.mjs` solve / `web/app.js` CHAPTERS)

## 3. コアロジック(検証済み・実装済み)

プロトタイプ段階で `logic.js` として実装・検証済み。以下のアルゴリズムが土台。

### 3-1. 状態と遷移
- 状態 = 各キャラの位置の組(+緩和ルール時はゴール固定フラグ。厳格では不要)
- 位置はインデックス表現 `index = y * w + x`
- 遷移: 各方向について、壁/外周ならその場維持(スキップ)、空きなら移動
- **ゴール進入制限**(1章): その手で全キャラが同時にゴールへ入るのでない限り、自分のゴールへ入ろうとするキャラはスキップ(ゴールを非同時時だけ壁とみなす)
- 全員がスキップする手 = 無意味手(状態が変化しない。手数に数えない)

### 3-2. ソルバー(BFS全探索)
状態空間が小さい(2部屋6x6で約170万状態、一瞬で全探索可能)ため、BFSで以下を機械的に算出:
- **解の有無**と**最短手数**
- **最短解の数**(=余詰めチェック。1なら一意=詰将棋的に美しい)
- **解に含まれる壁当て手の数と位置**(=妙手の保証とハイライト用)
- **部屋別の独立最短手数**(各部屋を単独で解いた場合)
- **同期コスト** = 同期解の最短手数 − 部屋別最短の最大値(同期制約が強いる遠回りの量)
- **最短解中の最小ズレ調整エピソード数**(episodes。連続する壁当て手のかたまりの数。難度の主軸。2-4)

### 3-3. 生成方式の転換(重要)
**実時間生成はやめ、オフラインで事前生成したプールJSONを配信する方式に確定。**

理由:
- プレイ時の採用率・BFS所要時間の制約が全部消える
- 3枚版(状態空間が部屋数の乗数で爆発)が現実的になる。生成が重くてもビルド時に回すだけ
- プレイ側はソルバーを積む必要がなく、JSONを読んで盤を描くだけ。スマホWebで一瞬
- デイリーパズルは「日付→問題ID」のマッピングだけで済む
- 生成段階で目視キュレーションや手直しを挟める(作品性の担保)

決定的乱数(mulberry32)を使い、**シード番号 = 問題の再現キー**。同じ logic バージョンなら seed から全部再現できる。

## 4. データスキーマ(確定)

事前生成プールの出力フォーマット。`nikenzume-pool-v1`。

```json
{
  "schema": "nikenzume-pool-v1",
  "generator": { "logicVersion": "1.0", "createdAt": "2026-06-11" },
  "puzzles": [
    {
      "id": 1042,
      "seed": 1042,
      "size": { "w": 6, "h": 6 },
      "rule": "strict",
      "rooms": [
        { "walls": "0,3,7,12,20", "start": 8, "goal": 31 },
        { "walls": "1,5,9,14,22", "start": 4, "goal": 29 }
      ],
      "solution": { "path": [0,0,3,3,2,1,2], "minMoves": 7 },
      "analysis": {
        "solutionCount": 1,
        "blockMoves": 2,
        "blockMoveIndices": [2,3],
        "soloMin": [5,6],
        "syncCost": 1,
        "episodes": 2
      },
      "tags": ["6x6", "strict", "7move", "unique", "hasMyoushu"]
    }
  ]
}
```

### フィールド定義
- `id`: 章編成・配信用の安定キー
- `seed`: 再生成・再検証用の種。データ破損時の保険・デバッグ用(「id 1042 を再現して」が一発)
- `size`: `{ w, h }`。両部屋で共通
- `rule`: 現状 `"strict"` のみ(厳格=同時到達)
- `rooms`: 部屋ごとの配列。**この配列の長さ = 部屋数**。3枚版は3要素にするだけで対応(スキーマ変更不要)
  - `walls`: 壁マスのインデックスをカンマ区切り文字列で(人間が読める・git diffで差分が見える形式を優先。圧縮は最後でよい)
  - `start`: キャラ初期位置のインデックス
  - `goal`: ゴールのインデックス
- `solution`:
  - `path`: 正解手順。方向の配列(`0=上, 1=下, 2=左, 3=右`)
  - `minMoves`: 最短手数
- `analysis`: 検証で出た値を全問焼き込む(再計算を避ける)
  - `solutionCount`: 最短解の数(1=一意)
  - `blockMoves`: 解に含まれる壁当て手の数
  - `blockMoveIndices`: 壁当て手が path の何手目か(妙手ハイライト用。ソルバーなしで光らせられる)
  - `soloMin`: 部屋別の独立最短手数 `[A, B]`
  - `syncCost`: 同期コスト
  - `episodes`: 最短解の中で最小になるズレ調整エピソード数(連続する壁当て手のかたまりの数)。**難度の主軸**(2-4)。多解問題では最も楽な経路で測る(最易解が難度を決める)
- `tags`: analysisから機械生成する派生情報。章を「タグでスライス」する運用の主役
  - 主要タグ: `"unique"`(解一意), `"hasMyoushu"`(壁当て手を含む), サイズ(`"6x6"`), 手数(`"7move"`), ルール(`"strict"`)

### プール運用方針(確定)
- **プールは全部入り1本。** ルール別・難度別に物理分割せず、配信時に tags でスライスする
- **緩い問題(解数が多い問題)も捨てない。** 生成段階のフィルタは最小(解の存在のみ)。品質判定は全部 tags に寄せる
  - 理由: データ生成は一度きりだが章編成は何度もやり直す。捨てると取り返しがつかない。残せば「やさしい章」に再利用できる
- **正解path(solution)は全問必ず持たせる。** ユーザーに見せるかどうかは実行時のUI判断。データ側に持たせ忘れると後で全問再生成になる
  - path があれば妙手ハイライト・答え再生・チュートリアル自動デモ・難度タグ付けが全部導出できる

### 正解pathを見せるかの判断(決定)
データには常に持つ。UIでは**「答えを再生する」ヒントとして見せる**(下記ヒント設計)。ただし**答えを見てもクリア扱いにはしない**(再生は見せるだけ→初形へ戻して自力で詰めさせる)ことでチート性を抑える。クライアントWebなのでチート耐性は本質的に限定的という前提は変わらず、デイリーで競わせる段になれば当該モードでのみ答え再生を制限/隠す方向で再検討する。

### ヒント・答えの設計(確定)
「分からないとき」の救済を3段で出す。本作の核(壁当て=空振りの妙手)を活かし、いきなり答えを見せず段階的に開く。

1. **Lv1 気づき**: 「壁当て(妙手)を◯回 含む」を提示(`analysis.blockMoves` 由来)。盤は見せず考え方の鍵だけ渡す
2. **Lv2 次の一手**: 今の局面の正解手を1つ、方向矢印で静かに一瞬示す(繰り返し可)
3. **Lv3 答え**: 全手順を自動再生(妙手ハイライト付き、`solution.path`)。再生後は初形へ戻す

- **次の一手は「軽量BFSをプレイ側に積む」**: プールの `path` は初形固定なのでプレイヤーが脇道に逸れると使えない。そこでヒント時にその場で最短を解く小さなBFSを積み、**今いるどの局面からでも**次手・残り手数を出す。状態空間は小さく一瞬(SPEC 3-3 の「ソルバーを積まない」は"生成用の重いソルバー"の話で、このヒント用BFSは別物として両立)。新ルール(非同時ゴール進入は反則)を枝刈りで守る。詰み筋を外した局面では「この局面からは詰みません」と案内
- UI: プレイ画面に静かな「ヒント」ボタン。押すと下からシートが出て Lv1 表示+「次の一手」「答えを再生する」。世界観(黒基調・引き算)を壊さない
- マネタイズ: 将来 Lv2/Lv3 をリワード広告と交換する想定(SPEC 6章)。**現状は機能のみ実装**し、広告接続は後(切れ目=広告枠は確保済み)

## 5. ビジュアル・世界観(確定)

### 方向性: ラグジュアリー(黒基調・発光ボール)
- 黒を基調とした空間に、少し光を帯びたボール(キャラ)が浮かぶ
- 引き算の美学。装飾を足さず、盤・壁・ボール・ゴールの4要素の質を上げ切る
- 参照: Monument Valley / Alto's Odyssey 系の静謐さ。Things 3 / Bear のミニマリズム精神
- **チープにしない。** ドット/レトロ路線は不採用(チープ化リスクが高く、洗練に高い技術が要るため)

### アニメーションの原則: 「演出」ではなく「情報」
動かす意味があるのは2つだけ。それ以外の装飾的演出(紙吹雪・派手なパーティクル・背景の動き)は世界観を壊すので入れない。

1. **キャラの移動**: マスからマスへスッと滑る。「どちらの部屋が動いて、どちらが動かなかったか」を伝える最重要の情報。100〜140ms / ease-out(速め寄り。思考ゲームはテンポ軽い方がよい)
2. **壁当て(妙手)**: 壁にぶつかって止まったキャラが進行方向に数pxめり込んで戻る(bump)。200ms前後。「入力は通ったがこの部屋では空振りした」を体で理解させる。両部屋で同時に起きることがあり、その非対称が一目で分かるのが価値

その他の状態演出:
- **ゴール到達**: 破線の円が実線に変わり、静かに塗りが満ちる。チェックは控えめ
- **クリア**: 盤上のボールが**金(最短達成)/白(クリア)に静かに発光**する。言葉(詰/CLEAR)は出さず、光だけで達成を伝える(**言語非依存** = クリアの climax を i18n から外す)。手数だけ下部に小さく添える。全体が一拍静止してから。派手にしない。音を付けるなら単発の硬い音
- **reduced-motion を尊重**(思考ゲームなのでアニメ無効でも完全に成立すべき)

### 発光表現の品質的注意
「光を帯びたボール」は安いと「ただのglowフィルタ」に見えてチープ化する。光を**ボール自体ではなくボールが盤に落とす効果**で見せること:
- 動くボールが通り道のマスをほのかに照らす
- 止まると静かに息づく(ゆっくりした明滅)
- ゴールに入ると光が満ちる

光を「状態の情報」として使うと、ラグジュアリーかつ意味のある演出になり、「アニメは演出でなく情報」の原則と一致する。

### 実装方式
- **素のCSS/JS で十分。** 動かす対象はグリッド上の円の transform だけ。Framer Motion 等は本作の動きの少なさに対して過剰
- 移動・bump・ゴールは CSS transition + transform で表現可能(プロトタイプで実証済み)
- 複数手の連続再生(答え再生)は async/await + transitionend で書ける(プロトタイプ実装済み)
- **文言(i18n)**: UI文言は `web/strings.js`(`window.NIKENZUME_STRINGS = {ja, en}`)に分離。`t(key, params)` で取得(英語の複数形は関数値で処理)、静的文言は HTML の `data-i18n` 属性から起動時に流し込む。**fetch せず JS 埋め込み**で `file://` でも動く。ロケールは `navigator.language` 自動 + `?lang=` 上書き(localStorage 永続)。タイトルは "Tsugai"、クリアは言語非依存(金/白発光)

## 6. マネタイズ設計(確定)

世界観の作り込みと広告は両立できる。本作はターン制パズルで「問題と問題のあいだ」に自然な切れ目があり、そこは思考が一区切りつく場所なので広告を差し込んでも没入を断ち切らない。

| 原則 | 内容 |
|---|---|
| **盤面は聖域** | プレイ中(盤面表示中)には広告を一切出さない。発光ボールが浮かぶ黒い空間は守る |
| **切れ目に置く** | 問題と問題のあいだ(クリア後・次の問題へ進む前)に広告。「詰み」の余韻のあと |
| **リワード型主軸** | インタースティシャルより、「ヒントを見る」「答えを再生する」等とリワード広告を紐づける。広告が"罰"でなく"交換"になる。詰将棋の「ヒントが欲しい」欲求と噛み合う |
| **買い切り除去** | 広告除去の買い切り(数百円)を必ず用意。ラグジュアリー路線のファン=世界観を金で守りたい層が有力な収益源 |

黒基調は広告の「間」を演出に変えられる。クリア後にすっと暗転して広告、終わると盤面がふっと浮かび上がる。暗転を世界観のリズムにできるのはラグジュアリー路線の強み(レトロ路線では使えない手)。
